1 # Silicon Labs Gecko Bootloader {#mainpage}
2 
3 # Introduction
4 
5 The Silicon Labs Gecko Bootloader is a common bootloader for all the newer MCUs and wireless MCUs (WMCUs) from Silicon Labs.
6 Key features of the bootloader are:
7 * Universal across families (MCU and WMCU)
8 * Supports image verification and encryption for
9  * Integrity
10  * Authenticity
11  * Confidentiality
12 * In-field upgradeable
13 * Configurable
14 
15 This document describes the requirements and the usage of the Gecko Bootloader.
16 
17 # Device Compatibility
18 
19 The Silicon Labs Gecko Bootloader is a customizable bootloader compatible with
20 
21 
22 * EFM32 Series 1 microcontrollers (Pearl Gecko 1, Jade Gecko 1 and
23  later)
24 * EFR32 Wireless Gecko SoCs
25 
26 The bootloader has a two-stage design, where the first stage's only job is to
27 enable upgrading of the main bootloader. This makes it possible to perform
28 in-field upgrades of the bootloader.
29 
30 # Requirements
31 
32 ## Flash Size
33 
34 The first stage of the bootloader takes up a single flash page. On devices with
35 2kB flash pages, like EFR32MG1, this means that the first stage takes 2kB.
36 
37 The size of the main bootloader is dependent on the functionality required. With
38 SPI flash storage and EBL encryption enabled, the main bootloader takes up 14kB
39 of flash, bringing the total bootloader size to 16kB.
40 
41 Silicon Labs recommends reserving at least 16kB for the bootloader.
42 
43 ## Capabilities
44 
45 Since the supports ECDSA signature verification of the
46 SHA-256 digest of the application for integrity and security reasons, the
47 bootloader can only run on devices that support hardware acceleration of these
48 operations. The bootloader is therefore only compatible with the EFM32 Series 1
49 microcontrollers and later, and EFR32 Wireless Gecko SoCs.
50 
51 # Application Interface
52 
53 The bootloader has an application interface that gets exposed through a function
54 table in the bootloader. The application interface provides APIs to make use
55 of functionality in the bootloader for storing and retrieving upgrade images,
56 verifying the integrity of said images, etc.
57 
58 In order to use the application interface in your application, include the
59 `api/btl_interface.h` header in your application, and add the following files
60 to the build:
61 
62 * `api/btl_interface.c`
63 * `api/btl_interface_storage.c`
64 
65 See the [Application Interface documentation](@ref Interface) for more
66 information.
67 
68 # Bootloader Components
69 
70 The bootloader itself consists of multiple parts:
71 
72 ## Core
73 
74 The bootloader core contains the main function of both bootloader stages. It
75 also contains functionality to write to the internal main flash, to perform a
76 bootloader upgrade, and to reset into the application flagging applicable reset
77 reasons. See the [core documentation](@ref Core) for more information.
78 
79 ## Driver
80 
81 Different bootloading applications require different hardware drivers for use by
82 the other components of the bootloader. See the [driver documentation](@ref Driver)
83 for more information.
84 
85 ## Plugin
86 
87 All parts of the bootloader that are either optional or swappable for different
88 implementations are implemented as plugins. Each plugin has a generic header
89 file, and one or more implementations. The current release contains plugins for
90 functionality like UART and SPI communication protocols, SPI flash storage,
91 internal flash storage, and different cryptographic operations. For more
92 information about the different plugins, see the [plugin documentation](@ref Plugin).
